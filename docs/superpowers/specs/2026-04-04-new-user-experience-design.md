# FrameHero CLI — New User Experience

Make `framehero` work reliably for any new user with any iOS app, regardless of navigation pattern (tab bar, sidebar, navigation stack).

## Problem

The CLI has three gaps that prevent new users from succeeding:

1. **Navigation only works for sidebar apps.** The current AppleScript accessibility approach traverses the Simulator's macOS accessibility tree to find sidebar toggle buttons. Tab bar apps, navigation stacks, and any non-sidebar layout fail silently or produce wrong screenshots.
2. **`framehero init` rejects non-interactive use.** It gates on `isatty()` and errors in CI or AI agent contexts. The underlying `UIDiscovery` uses a non-existent `simctl ui booted describeSiblings` command.
3. **No pre-flight validation.** If no simulator is booted, the wrong device is running, or the app isn't installed, users get cryptic `simctl` errors deep in the capture flow.

## Design

### 1. XCUITest-based navigation

Replace the AppleScript accessibility approach with Apple's XCUITest framework. `TestGenerator` already produces the right test code — `app.descendants(matching: .any)["Map"].tap()` works for any app layout.

**Throwaway Xcode project:**

`CaptureCommand` creates a minimal, temporary Xcode project at runtime:

```
/tmp/framehero-<uuid>/
  FrameHeroCaptureTests/
    FrameHeroCaptureTests.swift    (generated by TestGenerator)
  FrameHeroCaptureTests.xcodeproj/
    project.pbxproj                (minimal, hardcoded template)
```

The project contains only a UI test target. It does not compile the user's app — `XCUIApplication(bundleIdentifier:)` attaches to an already-installed app on the simulator.

**Execution flow:**

1. Generate test file from `framehero.yml` screens via `TestGenerator`
2. Write minimal `project.pbxproj` from a hardcoded template string
3. Run: `xcodebuild test -project <path> -scheme FrameHeroCaptureTests -destination "platform=iOS Simulator,name=<device>" -resultBundlePath <path>`
4. Extract screenshots from `.xcresult` via `ResultCollector`
5. Copy screenshots to output directory, organized by locale
6. Clean up temp directory via `defer`

**Locale handling:**

For each locale, set the simulator locale via `simctl spawn booted defaults write`, then run the XCUITest. The test relaunches the app per test method, picking up the new locale.

**Fallback:**

If `xcodebuild` is not available (checked at startup), error with: `"Xcode Command Line Tools required. Run: xcode-select --install"`. No silent fallback — XCUITest is the only navigation path.

For configs with only `launch` actions (no tap/navigate), skip the XCUITest project entirely and use direct `simctl io booted screenshot` — simpler and faster.

### 2. Dual-mode `framehero init`

Remove the `isatty()` gate. Detect terminal mode and behave accordingly.

**Interactive mode** (human in terminal, `isatty() == true`):

1. Boot simulator if needed, launch app
2. Discover screens via AppleScript accessibility scan of the Simulator window (find buttons, infer tab/sidebar items)
3. Display discovered screens, prompt user to select which to capture
4. Prompt for locales (default: `en-US`)
5. Write `framehero.yml`

**Non-interactive mode** (AI agent / CI, `isatty() == false`):

1. Launch app on booted simulator
2. Discover screens via same accessibility scan
3. Include all discovered screens (no prompt)
4. Use default locale `en-US`
5. Write `framehero.yml` and print its path to stdout

**Fix UIDiscovery:**

Replace the broken `simctl ui booted describeSiblings` call with the working AppleScript approach from our current `TestRunner` — scan the Simulator window's accessibility tree for buttons. This is acceptable for discovery (best-effort, informational) even though it's not reliable enough for navigation.

**Self-documenting output:**

The generated `framehero.yml` includes YAML comments above each section:
- `# App to capture` above `app:`
- `# Screens to capture. Actions: launch, tap "Label", navigate "A" > "B"` above `screens:`
- `# Locales to capture (BCP 47 codes)` above `locales:`

This makes the file editable by humans and AI agents without needing to reference docs.

### 3. Simulator validation

Both `init` and `capture` run pre-flight checks before doing any work:

**Check 1 — Simulator booted:**

```
xcrun simctl list devices booted
```

If no devices booted, error: `"No simulator booted. Run: xcrun simctl boot \"iPhone 16 Pro Max\""`

**Check 2 — Device match:**

Compare config's `simulator` field against booted device names. If no exact match:
- Interactive: warn and ask to use the booted device
- Non-interactive: use the booted device, print a note to stderr

**Check 3 — App installed:**

```
xcrun simctl listapps booted
```

Check for the bundle ID. If not found: `"App <bundle-id> not found on simulator. Build and run it from Xcode first."`

All errors are actionable — they tell the user exactly what command to run to fix the problem.

### 4. Error handling

**`xcodebuild` missing:** Check at startup with `which xcodebuild`. Error: `"Xcode Command Line Tools required. Run: xcode-select --install"`

**XCUITest build failure:** Surface `xcodebuild` stderr output. Common cause: Xcode version mismatch or missing simulator runtime.

**XCUITest element not found:** The generated test uses `element.waitForExistence(timeout: 5)` with `XCTAssertTrue`. If the element doesn't exist (wrong label in config), the test fails with a clear message: `"Could not find element 'Map'"` — this surfaces through the `.xcresult` and `ResultCollector`.

**Cleanup:** `defer` block removes temp Xcode project directory after capture, whether it succeeds or fails.

## Files changed

| File | Change |
|------|--------|
| `Sources/Commands/CaptureCommand.swift` | Rewrite to use XCUITest project generation + `xcodebuild test` |
| `Sources/Capture/TestRunner.swift` | Simplify to launch-only fallback (remove AppleScript navigation) |
| `Sources/Capture/TestGenerator.swift` | No changes needed — already generates correct XCUITest code |
| `Sources/Capture/ResultCollector.swift` | No changes needed — already extracts from `.xcresult` |
| `Sources/Capture/XcodeProjectGenerator.swift` | **New** — generates minimal `.xcodeproj` with UI test target |
| `Sources/Capture/SimulatorValidator.swift` | **New** — pre-flight checks (booted, device match, app installed) |
| `Sources/Commands/InitCommand.swift` | Remove `isatty` gate, add non-interactive mode |

| `README.md` | Update requirements, quick start, troubleshooting |

## Out of scope

- Caching compiled test bundles (optimization for later)
- Android / non-iOS support
- Deep link navigation (beyond tap/navigate)
- Automatic screen discovery without launching the app
